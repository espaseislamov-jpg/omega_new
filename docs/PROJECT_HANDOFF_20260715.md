# Технический handoff проекта Omega на 15.07.2026

Этот документ предназначен следующему разработчику/боту. Он описывает не только
структуру файлов, но и принятые решения, опасные места и обязательный порядок
проверки. Перед изменениями всё равно нужно посмотреть `git status`, последний
коммит ветки `work` и актуальный diff: документ фиксирует состояние на момент
написания, а не заменяет Git.

## 1. Что делает программа

Программа читает многопробный CSV-экспорт ChemStation, последовательно
обрабатывает каждую хроматограмму, назначает обнаруженные пики жирным кислотам,
интегрирует площади и рассчитывает итоговый Omega-3. Пользовательский сценарий:

1. оператор открывает CSV с батчем;
2. программа без параллельной нагрузки последовательно считает все пробы и
   показывает окно прогресса;
3. после окончания открывается GUI с общей хроматограммой, четырьмя
   нормированными фрагментами, списком проб и таблицей пиков;
4. судья даёт одно из трёх операторских решений: `ГОТОВО`, `ПРОВЕРИТЬ`, `СТОП`;
5. границы выбранного пика можно перетащить мышью или ввести как Start/End RT;
6. Batch Results копируется в буфер табличным текстом; номер образца там — часть
   имени после `On_`, без `.D`.

Предметно наиболее важны кластеры C18, C20 и C22. Итог особенно чувствителен к
C20:5 (EPA), C22:6 (DHA), C22:5 (DPA), C22:4 и к знаменателю C18.

## 4. Production pipeline по шагам

`omega_core.pipeline.process_batch()`:

1. Chebyshev baseline (`signal.add_baseline`).
2. `process_from_baseline()`:
   - сглаживание/производные;
   - поиск первичных пиков;
   - `matching.match_targets_to_peaks()`;
   - адаптер ChromatoPy, если применим;
   - `clusters.refine_cluster_matches()`;
   - аннотация RT-профиля;
   - расчёт Omega.
3. `metrics.annotate_result()` добавляет cluster quality и confidence.
4. При определённых формах C22 пробуется ASLS shape fallback.
5. При неполном cluster quality может пробоваться arPLS fallback.
6. Если финальный судья выдаёт **структурный** STOP (`missing_key_peak` или
   `duplicate_peak_assignment`), выполняется ровно один дополнительный полный
   проход:
   - альтернативный baseline (обычно ASLS);
   - `strict_matching=True`;
   - запрещён дальний nearest fallback для reliable RT;
   - soft RT ограничен 0.080 минуты;
   - order-only цель нельзя назначить первому случайному пику, её может назначить
     только guarded cluster override;
   - проверяются ключевые площади, уникальность peak ID, порядок и близость
     C18/C20/C22 к RT-профилю.
7. Повтор принимается только если структурный STOP исчез и строгая sanity-check
   пройдена. Иначе сохраняется первый результат и оператору явно пишется, что
   автоперепроверка не помогла.

Важно: это не бесконечный подбор вариантов и не oracle. Ровно один повтор не
использует ручное значение и не «натягивает» результат.

## 5. Почему возникало назначение чужого пика

В обычном `matching.match_targets_to_peaks()` reliable-цель при отсутствии пика
в окне 0.035 минуты могла брать ближайший свободный пик, пока расстояние не
превышало 0.3 минуты. Это исторический fallback, полезный на части старых
хроматограмм, но слишком разрешающий для автоматического исправления STOP.

Обычный первый проход пока оставлен обратно совместимым. Строгий повтор никогда
не использует такой дальний fallback. Если он не может подтвердить назначение,
пик остаётся ненайденным, STOP сохраняется и решение принимает оператор. Лучше
честный STOP, чем площадь химически постороннего пика.

На пяти известных структурных STOP-пробах до полного regression run:

- `06072026 O22`: повтор не нашёл C22:6, оставлен STOP;
- `07072026 O4`: повтор не нашёл C22:6, оставлен STOP;
- `07072026 O28`: строгий ASLS-повтор восстановил назначение, Omega приблизился
  к ручному значению (ошибка 0.260 → 0.096);
- `11072026 O44`: повтор не нашёл C22:6, оставлен STOP;
- `11072026 O47`: повтор не нашёл C22:6, оставлен STOP.

## 6. Границы, площадь и новый визуальный footprint

Расчётные поля — `integration_start_x`, `integration_end_x`, `area`. Они должны
оставаться единственным источником Omega. Ручная интеграция пересчитывает площадь
по `y_corrected`, обрезанному снизу нулём, через `numpy.trapezoid`.

Попытка 1c6699b двигала сами расчётные границы к минимумам сглаженного сигнала.
Она отменена. Вместо этого `ChromatogramApp._visual_peak_footprint_bounds()`:

- не меняет DataFrame и Omega;
- ищет ближайшую видимую «ногу» пика максимум на 0.024 минуты наружу;
- не пересекает midpoint близкого соседнего target apex;
- рисует дополнительный светлый слой по сглаженной кривой;
- поверх него более тёмным цветом остаётся фактически рассчитанная область и
  её реальные вертикальные границы.

Таким образом оператор видит полную форму пика и одновременно видит, какая часть
реально посчитана. Это намеренное разделение display geometry и numeric geometry.

Проверочный запечатанный raw-кейс `14072026 O7_905564655906.D` (без чтения
ручного ответа) дал, например:

- C18:2N6C: расчёт 7.5600–7.5873; footprint 7.5447–7.5873;
- C22:6: расчёт 9.2130–9.2403; footprint 9.2040–9.2430.

Именно такие вертикальные обрывы были показаны пользователем на скриншотах.

  1100×700.
- Верхняя панель разделена на действия и информационную строку, поэтому меньше

- В sidebar список батча и детали находятся в вертикальном `ttk.Panedwindow`.
- Главный список проб теперь показывает образец, Omega и однозначное решение.
- Batch Results показывает то же решение; строки цветные:
  - зелёный: `ГОТОВО — правка не нужна`;
  - жёлтый: `ПРОВЕРИТЬ` + коды пиков;
  - красный: `СТОП — переинтегрировать`;
  - серый: ещё не рассчитано.
- После принятого строгого повтора зелёная строка говорит `перепроверено
  автоматически`.
- Во время перетаскивания ручной границы больше не вызывается полный
  `axis.clear()`/перерендер всех пяти графиков на каждое событие мыши. Раз в
  ~16 мс обновляются только красный fill и две boundary line, через `draw_idle`.
- На отпускании мыши выполняется один настоящий пересчёт, границы привязываются
  к реальным x-сэмплам, а масштаб графиков сохраняется.

## 8. Судья: смысл и ограничения

Production judge находится в `omega_core.metrics.classify_high_error_risk()`.
Он reference-free: в GUI у него нет ручного ответа. Цель — отправить на ручную
проверку все известные ошибки `abs(error) > 0.5`, а не точно предсказать ошибку.

Есть два уровня:

- score >= 95: структурный STOP (нет ключевого пика или конфликт назначения);
- score >= 85: геометрический high-risk C20/C22, оператор проверяет отмеченные
  пики.

Низкий общий confidence сам по себе больше не создаёт STOP. `geometry_score`
отделён от отображаемого `score`: предупреждение судьи может понизить видимый
балл, но не имеет права переключать baseline/model и менять Omega.

Надписи специально переведены из абстрактных баллов в действие. Это не
математическая гарантия на будущих батчах. После добавления новых ручных дат
нужно заново проверять 100% recall и не сужать thresholds по одному примеру.

## 9. Запечатанный кейс 12072026 / старое имя 14072026

Пользователь уточнил: набор, ранее обозначенный `14072026`, фактически относится
к дате записи `12072026`; `14072026` было датой его обработки. Это один и тот же
sealed holdout. Пользователь знает ручные результаты, разработчик не должен
использовать их для порогов, выбора варианта или настройки алгоритма.

- `scripts/export_quality_dataset.py` жёстко исключает дату через
  `SEALED_DATES = {"12072026", "14072026"}`, чтобы защищать и старое, и новое
  имя файла.
- Raw CSV можно использовать только для проверки работы GUI, отсутствия crash и
  reference-free геометрии, как в footprint-примере выше.
- Нельзя открывать/парсить ручной XLSX этого кейса и нельзя просить пользователя назвать
  правильные значения до финальной проверки.

## 10. Regression corpus и особые соответствия

Основной измеренный corpus: 411 хроматограмм из 17 незапечатанных дат. Полный
экспорт запускается последовательно и занимает несколько минут:

```powershell
python scripts\export_quality_dataset.py --output artifacts\quality_current.csv
```

Финальный прогон этой итерации: 34 ошибки выше 0.5, судья поймал 34/34,
отмечено 105/411 проб, из них 71 ложное предупреждение; MAE 0.23157, RMSE
0.33709. По сравнению со стабильным расчётом изменился ровно `07072026 O28`,
его ошибка уменьшилась с 0.260 до 0.096; новых переходов через 0.5 нет.

Особый случай `03072026`: первая инъекция O1 пустая; ручная таблица сохранила
исходную позицию, а ChemStation сдвинул непустые имена. В
`omega_regression.resolve_batch()` эта дата сопоставляется по позиции после
пустой инъекции. Не заменять это на обычный sample ID без повторного ручного
аудита.

`artifacts/`, `regression_outputs/`, debug plots и сборочные каталоги локальные и
игнорируются Git. Не коммитить сгенерированные полные feature-таблицы без
отдельной причины.

## 11. Обязательная проверка после изменения расчёта

Минимум:

```powershell
python -m py_compile New_idea.py omega_regression.py omega_core\*.py
git diff --check
```

Для изменения matcher/integrator/metrics обязательно:

1. полный export всех 411 незапечатанных проб;
2. количество `abs_error > 0.5`;
3. применение `omega_regression.safety_judge()` ко всем строкам;
4. `missed == 0`;
5. отдельный контроль, что строк 12072026 и 14072026 в dataset ровно 0;
6. сравнение MAE/RMSE и списка новых переходов через 0.5 с предыдущим стабильным
   dataset;
7. smoke-test известных структурных STOP и GUI sample 1407 без ручных ответов.

Для GUI дополнительно проверить:

- окно можно уменьшить/увеличить, sash двигаются;
- списки не теряют выбранную пробу;
- все строки после preload получают цвет и действие;
- drag не сбрасывает zoom;
- во время drag меняется только overlay, после release площадь/Omega обновляются;
- Batch Results копирует номер, значение и решение.

## 12. Известные риски и что делать дальше

1. После чистки `New_idea.py` содержит только GUI. Численные изменения теперь
   делать в соответствующем модуле `omega_core`, а не возвращать алгоритмы в
   точку входа.
2. Judge оптимизирован под исторический recall и даёт заметные ложные
   предупреждения. Пользователь предпочитает лишнюю проверку пропущенной ошибке,
   но интерфейс обязан показывать действие, а не непонятный балл.
3. Принятый strict retry пока помогает не каждому missing C22. Не запускать
   третий/четвёртый подбор: один неуспешный повтор должен завершаться STOP.
4. Визуальный footprint не входит в площадь. Нельзя незаметно использовать его
   в `compute_omega`; для изменения чисел нужен отдельный A/B regression.
5. Полноценная нейросеть качества отложена до нескольких тысяч размеченных проб.
   На текущем объёме небольшой MLP переобучался и не давал требуемого надёжного
   recall на новых группах.
6. Не пересобирать EXE на каждой итерации. Пользователь отдельно попросит новый
   билд, когда код стабилизируется.
